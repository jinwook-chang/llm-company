# llm-company-wiki

기업용 원시 문서(Raw files)를 분석하여 Obsidian 스타일의 LLM 기반 내부 위키를 구축하는 도구입니다.

## 🚀 개요 (Overview)

이 프로젝트는 다양한 형식의 기업 내부 문서(PDF, Excel, Markdown 등)를 처리하여, LLM을 통해 지식을 추출하고 서로 연결된 지식 베이스(Wiki)를 자동으로 구축합니다. 결과물은 Obsidian과 같은 도구에서 바로 사용할 수 있는 Markdown Vault 형태로 제공됩니다.

## 🛠️ 설치 및 설정 (Setup)

### 1. 의존성 설치
이 프로젝트는 `uv`를 사용하도록 구성되어 있습니다.
```bash
# uv 사용 시
uv sync

# 또는 pip 사용 시
pip install .
```

### 2. 환경 변수 설정
`.env.example` 파일을 복사하여 `.env` 파일을 생성하고 필요한 API 키를 설정합니다.
```bash
cp .env.example .env
```
주요 설정 항목:
- `OPENAI_API_KEY`: OpenAI API 사용 시
- `LLM_WIKI_PROVIDER`: 사용할 LLM 프로바이더 (openai, vertex, azure_openai 등)

### 3. 프로젝트 설정
`wiki.config.toml` 파일에서 경로 및 LLM 모델, 동시성(concurrency) 등을 설정할 수 있습니다.

## 🏃 실행 방법 (How to Run)

설치 후 `llm-wiki` 커맨드를 사용하여 각 단계를 실행할 수 있습니다.

### 전체 프로세스 실행
전처리부터 위키 구축까지 한 번에 실행합니다.
```bash
llm-wiki all
```

### 단계별 실행
1. **전처리 (Preprocess):** Raw 파일을 Markdown으로 변환
   ```bash
   llm-wiki preprocess
   ```
2. **위키 구축 (Build):** 요약 생성 및 페이지 생성
   ```bash
   llm-wiki build
   ```
3. **정제 (Refine):** 중복 페이지 병합 및 링크 수정
   ```bash
   llm-wiki refine
   ```

## 🔄 프로세스 동작 방식 (Process Workflow)

위키 구축은 총 5가지 주요 단계를 거쳐 진행됩니다.

### 1. 전처리 (Preprocess)
- **목적:** 다양한 포맷의 원시 파일을 LLM이 이해하기 쉬운 Markdown으로 표준화합니다.
- **방식:** 
  - 텍스트/Markdown 파일: 직접 읽기(Direct reading).
  - PDF/복잡한 파일: LLM을 통한 추출 또는 `docling` 라이브러리를 사용한 폴백(Fallback) 처리.
- **결과:** `preprocessed/` 디렉토리에 소스 메타데이터를 포함한 `.md` 파일들이 생성됩니다.

### 2. 요약 생성 (Summarize)
- **목적:** 위키 페이지 생성 시 주변 맥락(Context)을 제공하기 위해 폴더 구조별 요약을 만듭니다.
- **방식:** 최상위 폴더부터 하위 폴더 순으로 계층적 요약을 생성하며, 하위 요약 생성 시 상위 요약 내용을 포함하여 일관성을 유지합니다.

### 3. 페이지 생성 (Generate)
- **목적:** 전처리된 개별 문서에서 핵심 컨셉을 추출하여 독립적인 위키 페이지를 만듭니다.
- **상세 동작:**
  - **컨텍스트 활용:** 단순히 파일 내용만 보는 것이 아니라, 해당 파일이 속한 폴더의 상위 요약 정보(`summaries`)를 함께 LLM에 전달하여 더 정확한 맥락을 파악합니다.
  - **멀티 컨셉 추출:** 하나의 소스 파일에 여러 개의 중요한 개념이 포함된 경우, LLM이 이를 분리하여 여러 개의 위키 페이지로 생성할 수 있습니다.
  - **한국어 우선 원칙:** 모든 본문 내용은 한국어로 작성하며, 기술 용어나 고유 명사는 필요 시 괄호 안에 영문을 병기합니다.
  - **슬러그 관리:** 파일명 충돌을 방지하기 위해 중복 시 `concept-2.md`와 같은 유니크한 슬러그를 자동으로 부여합니다.

### 4. 정제 및 병합 (Refine)
- **목적:** 서로 다른 소스에서 생성되었지만 동일한 개념을 다루는 중복 페이지들을 하나로 합치고 연결을 최적화합니다.
- **상세 동작:**
  - **다층 그룹화 (Multi-layer Grouping):**
    - 1차: 제목, 별칭(Alias), 슬러그의 완전 일치 여부 확인.
    - 2차: 임베딩(Embedding) 기반의 코사인 유사도 분석(0.8 이상)을 통해 유사 후보군 추출.
    - 3차: 추출된 후보군을 LLM에 전달하여 실제로 동일 엔티티인지 최종 판정.
  - **지능적 병합:** LLM이 여러 소스 페이지의 내용을 중복 없이 통합하며, 가장 적절한 정전 제목(Canonical Title)을 결정합니다. 이때 모든 원본 소스 경로는 `source_paths` 메타데이터에 보존됩니다.
  - **내부 링크 업데이트:** 병합으로 인해 사라진 페이지를 가리키던 위키 링크들을 새롭게 결정된 정전 제목으로 일괄 수정합니다.

### 5. 링크 해결 (Resolve)
- **목적:** 위키 내의 모든 `[[내부 링크]]`가 유효한 페이지를 가리키고 있는지 검증하고 최적화합니다.
- **상세 동작:**
  - **유연한 매칭 (Variant Matching):** 단순히 제목이 같은지만 보는 것이 아니라, 단수/복수형, 괄호 제거, 특정 접미사(Business, Division 등) 유무를 고려한 다양한 변체를 생성하여 매칭을 시도합니다.
  - **정전화:** 별칭(Alias)으로 연결된 링크를 실제 페이지의 제목으로 변환하여 Obsidian 내에서의 연결성을 강화합니다.
  - **미해결 링크 리포팅:** 매칭되는 페이지를 찾지 못한 링크들은 `.wiki_build/reports/unresolved_links.md` 파일에 목록화하여 사용자가 추후 보완할 수 있도록 합니다.

## 📂 프로젝트 구조

- `src/llm_wiki/`: 코어 로직 (전처리, 요약, 생성, 정제 등)
- `raw/`: 원시 데이터 저장소
- `preprocessed/`: 전처리된 Markdown 파일 저장소
- `vault/`: 최종 생성된 Obsidian Wiki 저장소
- `.wiki_build/`: 빌드 로그 및 리포트 저장소
